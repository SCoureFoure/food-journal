## [2026-05-14] ai-scout — Full Audit

**Rules audited:** yesterday, this_morning, leftovers, same_as_before, days_ago, named_day, meal_breakfast, meal_lunch, meal_dinner, meal_snack

**New patterns added:**

- `two_days_ago` rule (new): `day before (?:last|yesterday)` — prevents "day before yesterday" from resolving to offset=1 instead of offset=2
- `yesterday`: added `the (?:night|day) before`, `the other night`
- `same_as_before`: added `same (?:old|one)`, `repeat`, `the (?:thing|one) i (?:had|ate|got)`, `like (?:what |what I |I )?had`
- `days_ago`: added `(?:few|couple) nights ago`, `a while (?:back|ago)`, `last week`

**Patterns deliberately NOT added:**

- `from (?:before|earlier)` in `days_ago` — too greedy; "leftovers from earlier" should stay as `this_morning`, not `days_ago`
- `what mom made` / family-reference phrases — too ambiguous, no clear dateOffset mapping

**Scenarios added:** 18 new rows in `scenarios_test.dart`

Categories added: slang (same old, repeat, the thing I had, like what I had), two-days-ago (day before yesterday, day before last), last week, multi-nights (couple nights ago, a while back), the other night, named-day+meal-type combos (wednesday breakfast, sunday dinner, saturday brunch), edge inputs (single word "again", "leftovers", repeated temporal, punctuation-heavy "leftovers!!!"), false-positive guard ("I usually have eggs")

**Priority boundary tests added:** 6 explicit priority tests in `reference_engine_test.dart`

- named_day overrides leftovers (confirmed)
- days_ago overrides this_morning (confirmed)
- two_days_ago overrides yesterday (new — required for "day before yesterday")
- named_day + meal_type both returned (confirmed)
- named_day with no weekday → matchRecent (confirmed)
- same_as_before alone → matchRecent, no offset (confirmed)

**Bugs found:**

1. Wrong comment in `reference_engine_test.dart` line 48: test description said "offset 4 (Monday May 11)" but Monday May 11 from Thursday May 14 is 3 days, not 4. The assertion was already correct (expects 3); only the test name string was wrong.
2. Worker `parse_meal` system prompt had no instruction for handling the "Recent meals:" context block — Gemini would see it but had no explicit rule about not treating historical meals as current food items.

**Fixed:**

- Comment in `reference_engine_test.dart`: "offset 4" → "offset 3"
- `worker/src/prompts.json` v1.2.0: added paragraph instructing Gemini to use "Recent meals:" only for resolving temporal references, not as food items to output
- New `two_days_ago` rule + priority handling in `buildQuerySpec()`
- All new patterns have corresponding test coverage (no untested rules added)

**Still open:**

- No integration test for rolling window pruning (requires real DB setup in test)
- No widget test for AI-off path (requires mock AiService + widget test harness)
- No test for Worker behavior with context block (requires network mock / Gemini mock)
- Quick-copy fallback UI when AI is off: `isReferential()` is gated by `_aiEnabled` — if this UI is ever built, `isReferential()` should run regardless of `_aiEnabled`
- Worker `MEAL_PARSER_URL` uses `gemini-flash-latest` floating alias — should pin to specific version if schema instability is observed
- Worker has no guard against `foods` coming back as an object instead of array (Flutter client would throw on `json['foods'] as List`)

**Real-world failures logged:** None yet — no prod usage logs available at time of audit.

**Patterns to watch:**

- Any new `days_ago` pattern that contains the word "earlier" risks collision with `this_morning`'s `\bearlier\b` rule. Always verify priority holds.
- `same_as_before` patterns that are very short (e.g. `\bagain\b`) will fire on inputs like "I had eggs again today" which is legitimately referential — this is correct behavior, but watch for false positives on non-referential uses like "stir-fry again" (new meal, not a past meal reference).
- `last week` in `days_ago` always resolves to offset=3 (hardcoded approximation). Named day override will correct this if user says "last tuesday" — but "last week's dinner" with no specific day stays at 3.

**Why:** Priority order in `buildQuerySpec()` (top to bottom):
1. `two_days_ago` (offset=2)
2. `yesterday` or `leftovers` (offset=1)
3. `days_ago` (offset=3)
4. `this_morning` (offset=0)
5. `named_day` overrides all of the above if a weekday name is found in input
