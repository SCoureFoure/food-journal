Generative QA — Testing Guide

What Is This Dashboard?

This dashboard measures how well an AI-powered feature honors its behavioral contracts — properties that must hold true across the entire range of real-world inputs.

Unlike traditional testing (which checks exact values: "given X, expect Y"), generative output validation checks properties: "adding more evidence must always increase confidence" or "changing capitalization must never change the output."

A system can pass every unit test and still fail these contracts — because unit tests check known inputs, while contracts check the shape of behavior across unknown ones.

The 5 Contract Types

MFT Minimum Functionality Tests

"Does the core feature work at all?"

Binary oracle tests — pass or fail, no grey area. These cover the canonical inputs every production feature must handle correctly. If any MFT fails, the feature cannot ship.

Example: "leftovers" must be detected as referential. "yesterday" must resolve to offset 1.

● CRITICAL — any violation blocks shipping
INV Invariance Tests

"Do irrelevant changes break it?"

Real users don't type perfect input. They write "LEFTOVERS" or "leftovers!!!" or "left overs". These perturbations must not change the detection result. A system that only works on clean input will fail constantly under real user load.

Tests: case mutation, punctuation stripping, synonym equivalence, word order, whitespace normalization.

● HIGH — brittleness; real user variation will trigger this
DIR Directional Tests

"Does adding more evidence move the result in the right direction?"

These don't assert exact values — they assert direction of change. More signals fired → confidence must increase. A named day added to vague input → matchRecent must drop. A violation means the system got less useful as you gave it more information.

DIR tests directly measure whether AI context injection is working. A DIR violation on the AI layer means the expensive context-injection pipeline is failing or counterproductive.

● HIGH — directional violation means AI feature is not improving with context
BDY Boundary Value Tests

"Do numeric edges behave correctly?"

Off-by-one errors live at boundaries. The ±10% macro tolerance must accept 10.0% and reject 10.1%. The rolling fingerprint window must prune at exactly entry #41, not #40 or #42. These tests sit at every defined numeric threshold — at N and at N+1.

● MEDIUM — edge cases affected; most users won't hit it until they do
SCN Scenario Table

"Does it handle what real users actually type?"

The scenario matrix is a living catalog of real-world phrases. Every row was added because a real user typed something like it, or a developer recognized a gap. Unlike MFT (which tests canonical inputs), SCN tests the long tail: slang, colloquialisms, abbreviations, regional phrasing.

A failing scenario means a specific phrase — one a real user will say — is broken. Treat each failing scenario as a bug report from a real user.

● MEDIUM — known real-world inputs are failing

Coverage gaps (⚠ 0 tests) are worse than failures. A failure tells you something is broken. A gap tells you nothing — the contract could be honored or violated, and you don't know which. A green 0/0 is not a passing score; it's silence.

What Do Violations Mean?

A violation is a broken behavioral guarantee — not just a failed assertion but a signal that the system cannot be trusted to behave correctly across all inputs of that type.

ContractViolation means…
MFTCore feature is broken. The fundamental promise of the feature is not being kept.
INVFeature is brittle. It works in a lab but breaks when real users type imperfect input.
DIRAI context is broken. Adding more information makes the system less accurate.
BDYSilent edge-case bug. The tolerance math or window logic has an off-by-one.
SCNKnown phrase is broken. A real-world input the system was explicitly designed to handle is failing.

The Test Tree

Tests are organized by file → group → test, mirroring the source structure. This makes it easy to understand what module a failure is in.

File headers show pass/fail counts at a glance. Group headers reflect the describe block in the test file — each group tests one specific property of the system. Individual tests are the actual assertions.

Click any test to open its full diagnostic in the right panel. The diagnostic shows the input (what was fed to the engine), the expected behavior (what the contract requires), and — for failures — the actual output with the full error.

Why Not Just Use Unit Tests?

Unit tests check that this specific input produces this specific output. They're necessary but not sufficient for AI-powered features because:

1. AI outputs are probabilistic. You can't assert exact values from a language model — you can only assert properties (direction, tolerance, format).

2. User inputs are infinite. You can't enumerate every phrase a user might type. Contracts let you assert properties that must hold across all of them.

3. Regressions hide in perturbations. A model update or rule change might make "LEFTOVERS" stop working even though "leftovers" still passes. INV tests catch this; unit tests don't.

Behavioral contract testing is the industry approach for validating generative and AI-adjacent features in production systems.
